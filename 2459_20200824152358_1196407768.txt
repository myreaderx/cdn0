href=\"https://twitter.com/khotyn/status/1152516986208837633\">推文原文</a>是这样的：</p> \n<blockquote> \n <p>写周报其实是一件压力很大的事情，其实周报是在逼迫你做每周的思考。</p> \n</blockquote> \n<p>当时看到后直击内心。因为做为一个自由工作者，我做得不太好，虽然我一直在尝试让自己变得更加自律，但我发现我缺少阶段性的反思。因为我要维护不少项目，于是我长期处在 multitasking 的状态下（这不好，不要模仿），在去年的这个时候，我已经养成记录工作时间这个习惯两年多了，但是，常常还是浑浑噩噩地过完一周又一周，时间花在哪里是有迹可循了，但是如何规划和管理时间，还是很随性，常常不知不觉一头裁进一些事务性的活（比如修 Bug）上，然后忘了时间。有时会偶发性回顾一下，然后感叹：我怎么又花这么多时间在 A 项目上了…类似这样顾此失彼的事情常常发生。简单地说，当时的我就是满足于花时间干活了，却没有去思考这些时间分配是不是合理。</p> \n<p>回顾我的前几份工作，当时公司的制度都是有要求写周报的，是相对比较简单的那种，列举一下做了哪些事即可。大多情况下应付一下就了事，并不会花太多脑力在上面。对个人而言，我之前一直觉得，我已经在用 GTD 工具了，把要做的任务根据不同的项目列好，然后一项项去完成，想回顾时一看过去一周完成了多少任务，就已经有「周报」的作用在里面了，所以这就够了。</p> \n<p>但这是完全不够的，GTD，只是标记你有没有完成某项工作的工具，它不会引导对工作的回顾和思考：</p> \n<ul> \n <li>过去一周工作的怎么样，是否按计划完成了？</li> \n <li>哪些任务超出了预期（低估了工作量），原因是什么？</li> \n <li>过去一周在创造型的工作上花了多久，在事务型的工作上花了多久？</li> \n <li>……</li> \n</ul> \n<p>当我意识到，我在 GTD + 时间记录上做得很好，却很少思考过之后这些问题的时候，我就觉得，是时候写周报，把这些问题都问一下自己了…于是，在去年的这个时间，结合看到这条推文，我引用并也发了一条<a href=\"https://twitter.com/tualatrix/status/1152552920031744000?s=12\">推文</a>，然后就开始写周报了，而不是仅满足于在 GTD 里面勾勾划划。</p> \n<p><img alt=\"after-writing-weekly-report-one-year.jpeg\" src=\"https://imtx.me/media/uploads/2020/07/after-writing-weekly-report-one-year.jpeg\" /></p> \n<h3>经历</h3> \n<p>刚开始逼自己写周报的过程是痛苦的，有时一周只是维护了项目、修了一些 Bug，写不什么所以然来，显得很单薄。有时周一（我定在周一写上周报）有事出去了，然后就不写了，最后写成了「双周报」（甚至月报）。但很快，我从写周报的过程中，找到了乐趣：</p> \n<ul> \n <li>周报是写给我自己写的，没有 Boss 看，于是我写了很多内心的想法在里面，有时是对自己的批评，也有对自己完成一件大事的表扬，我忠实地记录了自己的思考路径，这是在公司上班时写周报时很难做到的；</li> \n <li>除了刚开始写周报只是记录在每个项目上花的时间外，渐渐地，我开始记录在工作种类上花的时间，我得以清晰地认知到，我在修 Bug 这类维护工作上，花了多久，我在学习和研究新东西上花了多久。假如我在第 N 周花了太多时间在修 Bug 上，那么 N+1 周我就有意识地要纠正回来，要多投入点时间在研发上。</li> \n <li>有段时间周报越写越详细了（刚开始我的周报内容还包括了下周计划），于是在 2020 年刚开始的时候，我决定把写可执行的「周计划」作为单独的内容，拆出来写，于是我又开始认真写「周计划」的过程。</li> \n <li>我还是会常常忘记写（毕竟不需要给人检查），于是我写了段代码，在每周一自动生成「周报」、「周计划」的草稿，躺在草稿箱里，我总会看到草稿，于是就会去写了…</li> \n</ul> \n<p>…</p> \n<p>我在过去的一年不断地优化围绕着写周报这件事，一周年之际，我可以说，继严格统计自己的工作时间后，我又养成了写周报（包括计划）这个习惯。我还会继续优化这个过程。</p> \n<h3>意义</h3> \n<p>「自我提升」是一个很艰难的过程，需要外力和内力的互相作用，最后就是让时间来证明了。写了一年周报后，我也只能说，我做到了记录并反思下每个阶段完成的工作，并开始展望下个阶段的工作。我不敢说，我通过这个习惯，我的创造力 Double 了，我的效率 Double 了，这几乎不太可能。</p> \n<p>但是，我感受到或多或少有那么一点掌控感了，也对自己有了更深的认知了：</p> \n<ul> \n <li>我知道自己做过了哪些计划，然后计划有没有完成，原因又是什么；</li> \n <li>我知道自己做哪类任务时，比较容易沉浸进去，但常常因此占用了过多时间；</li> \n <li>我也知道了自己比较抵触哪类任务，常常连续好几周不停地 postpone；</li> \n <li>写了一段时间的周报后，我发现自己的长期计划有点模糊，并不确定，于是又开始把长期计划通过文字写下来；</li> \n <li>…</li> \n</ul> \n<p>有些工作是不得不做的，抵触并不意味着，我做不到，或者做不好，只是惯性太大，我习惯性回避罢了。我相信我清晰地认知到这些后，我就有可能去挑战和改进它，就像养成写周报这个习惯。</p> \n<h3>展望</h3> \n<p>未来我应该会继续把这件事情做下去。其实写周报就像写日记一样，就是一个直面自己的过程。写日记除了记录下当下的事件和思考外，可能并没有直接的收获，但是，当回顾的时候就会看到：原来最近我纠结的一个点，之前也纠结过，我没有试着去改进它所以它又发生了；又或者，我已经走出了之前的一个障碍…</p> \n<p>写到这里，我觉得本文已经跳出了「写周报」这个话题，有点宏大了…差不多就此打住吧。</p> \n<p>我相信，在一年后回顾所这篇总结，很多东西会发生变化，我会有新的认知和习惯，甚至会推翻之前的一些想法。我觉得这是好事：看到自己在过去某个时间节点的认知并和当下做比较，不断去调整、变化和提升。这就是记录的意义。</p> \n<hr /> \n<p align=\"center\"><strong>本站架设于 <a 备份中恢复两种方式。两者都不错，但都不是很方便：前者需要手动备份至电脑上再恢复，步骤繁琐；后者从 iCloud 备份恢复受网络影响较大，通常比较费时间。这次设置新手机，我采用了新的直接「手机互传」恢复的方式。不仅操作简单，速度也非常快，只花了一个多小时就将用了一百多 GB 的 iPhone X 一模一样的恢复到了新的 iPhone 11 Pro Max 上。</p> \n<p>同样方便的还有 <li>多了一个位置可以放额外功能，于是我新增了 Spotify；</li> \n <li>天气温度不仅是显示当前气温，还会显示今天的最低温和最高温；</li> \n <li>「在意空气」除了能显示空气质量指数以外，还会通过圆环信息指示当前所处的等级；</li> \n <li>日历信息除了显示今日有没有行程之外，还有完整的月份和日期信息。</li> \n</ul> 的按钮就可以解锁。可能有人会问，这真的会比输入密码、或指纹更方便吗？当然。比如我在工作的时候，很多时候都是合上电脑用外接显示器来工作的，这时就无法用指纹，电脑的解锁密码又比较复杂，没有按 Apple Watch 两下来的方便快捷。</p> \n<p><img alt=\"重新体验 iPhone：解锁电脑，解锁更多可能之二.jpg\" \n<p>但就是这样的一个小 demo，也有很严重的 Bug。先说 macOS 端：</p> \n<p>macOS 版本，跑起来以后，直接点 Random Select，是可以正常工作的，你会发现 List 左侧选中的项目，在随机跳动。</p> \n<p>但是，一旦你手动选择过一个项目后，再点 Random Select，就不管用了。</p> \n<p>而 iPadOS 下面，无论怎么点，都是不管用的…</p> \n<p>昨天遇到这个问题后，我一直在想，我哪里做错了，是不是有什么条件未考虑到。今天我再想想，抛开 SwiftUI 还不成熟的话题，这就是 SwiftUI 的局限性，或许，我要做的任务，根本就不应该来用 SwiftUI 来完成。</p> \n<h3>SwiftUI 的局限性是什么？</h3> \n<p>SwiftUI 是什么？它是一套跨平台的框架，我上述代码，也完美的验证了同样的代码，可以跑在 macOS 和 iOS 上，并且都非常 Native——除了行为没有达到我的预期。</p> \n<p>作为同时写过 macOS 和 iOS App 的开发者，我思考这个问题出现的原因，首先就是 AppKit 和 UIKit 下 API 和预期行为的大不同。</p> \n<p>假如我们分别用平台对应的 API 去达到「选择列表行」的效果的话，那么要用到的 API 分别是：</p> \n<ul> \n <li>macOS：NSTableView 的 <code>func selectRowIndexes(_ indexes: IndexSet, byExtendingSelection extend: Bool)</code></li> \n <li>iOS：UITableView 的 <code>func selectRow(at indexPath: IndexPath?, animated: Bool, scrollPosition: UITableView.ScrollPosition)</code></li> \n</ul> \n<p>这两个 API，不仅名字不一样，参数数量不一样，连参数类型都不一样。通过各种排列组合，有太多调用方式和结果了。</p> \n<p>我不知道 SwiftUI 底层是怎么实现的，但是据目前了解到的信息，除了底层绘制的相关 View，大多数 SwiftUI 的控件，都是去调用对应平台的原生控件的，就像 List 在 macOS 下是 NSTableView，在 iOS 下是 UITableView。</p> \n<p>于是，要达到预期的 List 选中（selection）的效果，我可以想象 SwiftUI 需要处理大量的逻辑，来使这个功能工作——根据目前的情况，一个小 demo 就暴露了 SwiftUI 目前的问题和不足。也就是我说的局限型。</p> \n<h3>SwiftUI 目前的使用场景</h3> \n<p>根据我有限的对 SwiftUI 的了解和实践，我觉得当前的 SwiftUI 适合的应用场景是：</p> \n<ul> \n <li>没有复杂逻辑的使用原生界面的列表型交互；</li> \n <li>需要高度自定义的、使用自定义绘制和动画的界面；</li> \n</ul> \n<p>尽管 SwiftUI 没有达成我简单的「随机选择」这么一个需求（我希望是我代码的问题），但是我依然相信 SwiftUI 会在特定场景下达到生产力的极大提升，只是目前真的很不成熟，Xcode 也比较让人失望。</p> \n<p>接下去能期待的一件事情，就是 Swift 5.2 的发布，它将极大的改进
