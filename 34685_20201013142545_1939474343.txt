style=\"font-weight: 400;\">We are, and always have been, hugely thankful for the work and mission that the Tor team brings to the world. To continue our support, we wanted to make our website and browser download accessible to Tor users by creating Tor onion services for Brave websites. </span><span style=\"font-weight: 400;\">These </span><a href=\"https://blog.torproject.org/more-onions-porfavor\"><span style=\"font-weight: 400;\">services</span></a><span style=\"font-weight: 400;\"> are a way to protect users’ metadata,</span><span style=\"font-weight: 400;\"> such as their real location, and enhance the security of our already-encrypted traffic.</span> <span style=\"font-weight: 400;\">This was desired for a few reasons, foremost of which was to be able to reach users who could be in a situation where learning about and retrieving Brave browser is problematic.</span></p> \n      <p><span style=\"font-weight: 400;\">We’ll go through the process of creating this setup, which you should be able to use to create your own onion service.</span></p> \n      <p><span style=\"font-weight: 400;\">To start the process we ‘mined’ the address using a piece of software called a miner: I chose </span><a href=\"https://github.com/lachesis/scallion\"><span style=\"font-weight: 400;\">Scallion</span></a><span style=\"font-weight: 400;\"> due to Linux support and GPU acceleration. Mining is the computationally expensive process of creating a private key to prove a claim on an onion address with a desired string. Onion (v2) addresses are 16 character strings consisting of a-z and 2-7. They end in .onion, and traffic to .onion domains does not exit the Tor network. </span><a href=\"https://blog.torproject.org/v2-deprecation-timeline\"><span style=\"font-weight: 400;\">V3 addresses</span></a><span style=\"font-weight: 400;\"> are a longer, more secure address which will provide stronger cryptography, which we will soon migrate to.</span></p> \n      <p><span style=\"font-weight: 400;\">In our case we wanted a string that started with ‘brave’ followed by a number. A six-character prefix only takes around 15 minutes when mined on a relatively powerful GPU (we used a GTX1080). The end result is a .onion address and a private key that allows us to advertise we are ready and able to receive traffic sent to this address. This is routed through a ‘tor’ daemon with some specific options.</span></p> \n      <p><span style=\"font-weight: 400;\">After we mined our onion address we loaded it up in EOTK. The </span><a href=\"https://github.com/alecmuffett/eotk\"><span style=\"font-weight: 400;\">Enterprise Onion Toolkit</span></a><span style=\"font-weight: 400;\"> is a piece of software that simplifies setting up a </span><a href=\"https://2019.www.torproject.org/docs/documentation.html.en\"><span style=\"font-weight: 400;\">Tor</span></a><span style=\"font-weight: 400;\"> daemon and </span><a href=\"https://openresty.org/en/\"><span style=\"font-weight: 400;\">OpenResty</span></a><span style=\"font-weight: 400;\"> (a Lua-configurable nginx-based) web server to proxy traffic to non-onion web servers. In our case we are proxying traffic to brave.com domains. One last piece was required to complete the setup: a valid SSL certificate.</span></p> \n      <p><span style=\"font-weight: 400;\">Without the certificate, upon starting&nbsp; EOTK for the first time, you’ll find that many web assets don’t load. This is due to using a self-signed SSL certificate. For some, this is acceptable. Many onion users are accustomed to seeing self-signed certificate warnings, however for the best experience a legitimate certificate from a CA is necessary. For now, the only certificate authority issuing certificates for .onion addresses is </span><a href=\"https://www.digicert.com/blog/ordering-a-onion-certificate-from-digicert/\"><span style=\"font-weight: 400;\">DigiCert</span></a><span style=\"font-weight: 400;\">. One is the traditional EV due diligence of contacting a representative of the organization that is on-file with DigiCert. Since the onion addresses don’t have the concept of DNS, TXT validation will be impossible. That leaves the only remaining option as the HTTP </span>\n      <a href=\"https://docs.digicert.com/manage-certificates/demonstrate-control-over-domains-pending-certificate-order/use-http-practical-demonstration-validation-method-verify-domain-control/\"><span style=\"font-weight: 400;\">practical demonstration</span></a>\n      <span style=\"font-weight: 400;\">. The request failed since we were still running an EOTK-generated self-signed certificate. EOTK has a feature to serve short strings such as those required for this process using the “hardcoded_endpoint_csv” </span>\n      <a href=\"https://github.com/alecmuffett/eotk/blob/master/docs.d/HOW-TO-INSTALL.md#proving-your-ownership-to-a-certificate-authority--hardcoded-content\"><span style=\"font-weight: 400;\">configuration option</span></a>\n      <span style=\"font-weight: 400;\">, but unfortunately it did not work due to the SSL redirect. I was able to modify the OpenResty configuration to move the configuration block responsible to the port-80 server section.</span>\n     </div> \n    </div> \n    <!-- .et_pb_text -->\n    <div class=\"et_pb_module et_pb_text et_pb_text_45  et_pb_text_align_left et_pb_bg_layout_light\"> \n     <div class=\"et_pb_text_inner\">\n      <pre>$ cat brave.conf\nset hardcoded_endpoint_csv ^/\\\\.well-known/pki-validation/fileauth.txt$,abc123\nhardmap secrets/${var.onion_domain}.key ${var.external_domain}</pre>\n     </div> \n    </div> \n    <!-- .et_pb_text -->\n    <div class=\"et_pb_module et_pb_text et_pb_text_46  et_pb_text_align_left et_pb_bg_layout_light\"> \n     <div class=\"et_pb_text_inner\">\n      <span style=\"font-weight: 400;\">After consulting with the author, I was told that the “force_http” EOTK option will fix this. Opening a chat session with a DigiCert rep solved this problem quickly though, especially after pointing out that DNS TXT validation is not possible, and providing a link to the .onion blog post referenced earlier.</span>\n     </div>\n    </div>\n   </div>\n  </div>\n </div></p> \n<p><span style=\"font-weight: 400;\">We had to reissue certificates a few times (requiring more rounds of human validation for the EV cert requirements) in order to add some SAN wildcard subjects for our various subdomains (for example *.brave.com will not match example.s3.brave.com).
