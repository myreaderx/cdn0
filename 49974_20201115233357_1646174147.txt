/>\n          \n        \n          \n        \n\n        \n      \n        </figure>\n      \n\n    \n  \n\n\n  \n\n\n<p>The write-ahead log is the durability feature that allows WiredTiger to survive a process or system crash. (MongoDB calls this the \"<a href=\"https://docs.mongodb.com/manual/core/journaling/\">journal</a>\".) Any thread writing data to WiredTiger first appends a record describing the write operation to the write-ahead log; in the event of a crash, any writes that were not persisted to the storage tables can be replayed from this log. The write-ahead log offers three durability modes. The strictest durability, ”full-sync”, guarantees the record is flushed to disk before returning, allowing the data to survive a system crash. The second level of durability, “write-only”, guarantees the record is written to the OS file system before returning to the user, allowing the data to survive a process crash. The least durable is “no-sync”, where the record is recorded in a buffer in memory but there is no guarantee it is immediately written to the file system.</p>\n<p>Traditional write-ahead logs use mutexes to coordinate their writes. On single-core architectures this works well, but when more cores are available, it wastes a lot of time making threads wait for a mutex. To achieve better parallelization in our write-ahead log, I had adapted a design from a research paper, \"<a href=\"https://infoscience.epfl.ch/record/170505/files/aether-smpfulltext.pdf\">Scalability of write-ahead logging on multicore and multisocket hardware</a>\", in the <em>Very Large Databases Journal</em>. Instead of writes being synced or written individually, many threads may copy their records into a single in-memory buffer, which can be written to the file system in a single call. The threads still have to wait for their desired level of durability to be achieved before they return, but the overall throughput of the write-ahead log is greater, because each thread isn’t making its own system call.</p>\n<p>Instead of mutexes, atomic operations provide for isolation of writes, allowing all the threads to run without locking. This design could not work with a mutex controlling access to the in-memory buffer — that would just re-introduce the contention to a different layer in the call stack.</p>\n<p>Traditional write-ahead logs also can’t offer a no-sync mode, but since WiredTiger already consolidated writes in memory, the ability to do so presented itself. It simply lets threads return without waiting for their writes to be written or synced. In this way, a set of logically related writes could be run much faster: client code could issue all but one write using no-sync and finish with a write-only or full-sync call.</p>\n<h2 id=\"the-first-hint-of-trouble\">The first hint of trouble</h2>\n<p>In the summer of 2015, Bruce Lucas, a Senior Technical Service Engineer at MongoDB, was helping make WiredTiger the default storage engine in MongoDB version 3.2. Bruce needs to have a high degree of expertise with the innards of MongoDB to support customers, so he was running experiments to study how WiredTiger performed under a wide variety of circumstances. One of those experiments, a workload consisting of many small inserts using a high number of threads per core, uncovered a case of negative scaling.</p>\n<p>Here are results from an AWS Linux box with 8 cores running the WiredTiger code from MongoDB 3.0.4:</p>\n<pre data-preserve-html-node=\"true\">threads    k ops/sec\n  8        278\n 16        379\n 24        232\n 32        158\n 48        126\n 64        118\n</pre>\n\n<p>Using <a href=\"http://poormansprofiler.org/\">Poor Man's Profiler</a>, which periodically invokes gdb to take full stack traces of all threads, Bruce was able to discern the bottleneck: busy-waits in WiredTiger's write-ahead log. As he dug deeper, he uncovered two mismatches between the design of the write-ahead log and the conditions under which it would run as a storage engine for MongoDB.</p>\n<h2 id=\"condition-mismatch-1-durability-level\">Condition mismatch #1: durability level</h2>\n<p>I first designed the write-ahead log before WiredTiger joined MongoDB, and I assumed that most applications using a write-ahead log would use write-only or possibly full-sync mode. This was the typical use case for customers at that time, who were using the WiredTiger library’s key-value store directly.</p>\n<p>MongoDB, however, is exactly the kind of system that benefits from issuing a series of related no-sync writes concluding with a full-sync write. WiredTiger structures data as key/value pairs within tables, and those tables are what all of MongoDB’s data structures are built upon — <a href=\"https://docs.mongodb.com/manual/core/databases-and-collections/#collections\">collections</a>, the <a href=\"https://docs.mongodb.com/manual/core/replica-set-oplog/\">oplog</a>, <a href=\"https://docs.mongodb.com/manual/indexes/\">indexes</a>, and various internal MongoDB meta-data are all WiredTiger tables. A single write to a MongoDB document results in many write calls into several WiredTiger tables, and from a client perspective, unless all of them persist, none of the rest matter. So even when a call to MongoDB specifies that the journal <a href=\"https://docs.mongodb.com/manual/reference/write-concern/#wc-j\">must be synced to disk before a write returns</a>, MongoDB can make use of the highly efficient no-sync mode for all but the last call. That now makes \"no-sync\" by far the most common case.</p>\n<h2 id=\"looking-deeper-the-wiredtiger-log-slot\">Looking deeper: the WiredTiger log slot</h2>\n<p>The prevalence of no-sync calls now puts WiredTiger’s write consolidation mechanism front and center. At it’s heart is a data structure called a \"<a href=\"https://github.com/wiredtiger/wiredtiger/blob/1c8aa34e3047f315a89d407a637c992e035d05ff/src/include/log.h#L80-L100\">slot</a>,\" using the terminology from the research paper. A slot encapsulates an in-memory buffer, related meta-data, and a special <code>int64_t</code> field called “<a href=\"https://github.com/wiredtiger/wiredtiger/blob/1c8aa34e3047f315a89d407a637c992e035d05ff/src/include/log.h#L81\"><code>slot_state</code></a>” which counts the number of bytes claimed in the buffer. We reserve the lower integers in <code>slot_state</code> for internal state values, the highest of which is called <code>READY</code>. Threads first <em>join</em> a slot that is <code>READY</code>, claiming a spot in its buffer with an atomic operation on <code>slot_state</code>, and when the slot is closed to further joins, they all copy their data and <em>release</em> their claim with another atomic operation on <code>slot_state</code>.</p>\n<p>WiredTiger maintains a <a href=\"https://github.com/wiredtiger/wiredtiger/blob/1c8aa34e3047f315a89d407a637c992e035d05ff/src/include/log.h#L158-L162\">pool of these slots</a>. To write a record into the log, a thread attempts to <em>join</em> the currently active slot if it is in the <code>READY</code> state. The first step is to determine if the record fits within the remaining buffer space by adding the record size to the <code>slot_state</code>. If the total fits within the buffer, the thread performs an atomic compare-and-swap of <code>slot_state</code> with the new total size. On success, the prior value of <code>slot_state</code> indicates this thread’s offset into the buffer and its new value reflects the correct offset for the next thread to join (or the final size of this buffer).</p>\n<p>The thread doesn’t write its payload yet! That write, and the associated bookkeeping, has to wait until the slot is closed to other joining threads — a task that belongs to the leader thread. So for now, it just waits, and watches <code>slot_state</code> for that closed indication.</p>\n<p>A joining thread that gets an offset of zero is designated the leader. It doesn’t sit around waiting for the slot to close, that’s <em>its</em> job. But first it prepares a new slot from the pool, to ensure one will be ready for the subsequent round of joins and writes. This is the one task in the entire process that requires a lock, and when it’s over, the leader atomically inverts the sign on <code>slot_state</code>. A negative <code>slot_state</code> indicates that the <code>slot</code> is closed to further joins — the sign the other writers have been waiting for. They already know where their data belongs in the buffer, so they can copy in parallel. Job done, they <em>release</em> the slot by atomically adding their record size to <code>slot_state</code>, which is now the <em>negative</em> of the total number of bytes left to copy into the buffer. When <code>slot_state</code> reaches zero, all copies are complete and that slot is fully prepared for writing. The last thread to release the slot sees this zero and takes charge of writing the buffer to the OS, owing nothing to the rest of the threads, which have already gone on their merry way.</p>\n\n\n\n\n\n  \n\n  \n\n\n\n  \n    \n      \n\n        \n          \n            \n              \n                <img class=\"thumb-image\" data-image=\"https://images.squarespace-cdn.com/content/v1/58c01cea29687fdfb7c37d2d/1493763354135-ZTKUBEWR7Y7HQPXTA1LE/ke17ZwdGBToddI8pDm48kA2MStDQoZ8TsVhOdZz9ancUqsxRUqqbr1mOJYKfIPR7LoDQ9mXPOjoJoqy81S2I8N_N4V1vUb5AoIIIbLZhVYxCRW4BPu10St3TBAUQYVKceADPfR09Ga36y34v15Tbupz64KQ7ohIffrpQ0Sh5Na6j4V7zcjOiktKTPaM-wfsD/01.png\" data-image-dimensions=\"1280x900\" data-image-focal-point=\"0.5,0.5\" alt=\"A slot in the READY state awaits joining threads. This was another assumption that I made in my original design, based on the expectation that WiredTiger would be embedded in specialized applications, not a general-purpose database.</p>\n<p>The availability of cores is important to threads that are waiting to copy records to the buffer, because they have to busy-wait until <code>slot_state</code> changes, consuming CPU cycles. Since I expected that the state would be updated very quickly and that there would be plenty of CPU available for the check, a busy-wait was safe when I wrote it. But MongoDB uses a different thread for every client connection. Some workloads, like those that perform a lot of tiny writes, create many more threads than cores.</p>\n<h2 id=\"the-bottleneck\">The Bottleneck</h2>\n<p>Individually, neither one of the condition mismatches would have been a disaster. But when there are a lot of threads and few cores, and when most of those threads are busy-waiting on a state change, we get a classic case of priority inversion. The thread in charge of managing <code>slot_state</code> and the slot pool ought to be top-priority, but all the threads that have joined a slot and are waiting for a change in <code>slot_state</code> have overwhelmed the CPUs. These threads do call <code>sched_yield</code>, but that doesn’t help much on Linux's scheduler.</p>\n<h2 id=\"stay-tuned-for-an-entirely-new-approach\">Stay tuned for an entirely new approach</h2>\n<p>We couldn’t just replace yielding with a mutex, because that would kill performance under all loads. He came up with a stand-alone prototype that held the promise of a massive improvement to our \"no-sync\" case.
