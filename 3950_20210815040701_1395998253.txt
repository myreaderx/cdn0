\n<p>举个例子，<code>Array.isArray()</code>方法可以判断一个对象是不是数组，此方法IE8浏览器并不支持。</p> \n<p>如果采用 polyfill 策略，则代码会是下面这样：</p> \n<pre>if (!Array.isArray) {\n  Array.isArray = function(arg) {\n    return Object.prototype.toString.call(arg) === '[object Array]';\n}</pre> \n<p>使用示意：</p> \n<pre>isArray([]) <span style=\"color:green;\">// 返回 true</span></pre> \n<p>同样是让所有浏览器支持某功能，但是，并不会采用和规范一致的API。</p> \n<p>也就是Ponyfill总是避免使用原生 API。</p> \n<h4>什么时候使用Ponyfill？</h4> \n<p>polyfill策略大部分时候都是很OK的，使用方便快捷，不需要关心各种细节。</p> \n<p>但是，并不是所有的时候都适合使用polyfill。</p> \n<ol> \n <li>有些原生API完全没法模拟，此时只能使用Ponyfill策略。例如本文一开始提到的 File System Access API，低版本浏览器完全搞不定。 <p>又例如 indexDB 能力，<code>history.pushState()</code>方法等，IE9浏览器根本没法模拟，只能使用 Ponyfill策略。</p></li> \n <li>有些原生API规范还没稳定，或者处于快速迭代中，或者是浏览器部分支持，此时走polyfill策略可能会有问题，影响未来策略。例如自定义元素v1规范和v2规范。</li> \n <li>有时候可能会通过对象存在与否判断浏览器，polyfill可能会影响逻辑。例如有业务代码使用 <code>if (![].map) {}</code> 判断是不是低版本IE浏览器，如果不小心引入了包含
