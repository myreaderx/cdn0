Historically, an Android view hierarchy has been represented as a tree of UI widgets. As the state of the app changes, the UI hierarchy needs to be updated to display the current data. The most common way of updating the UI is to walk the tree using functions like <a href=\"https://android.devsite.corp.google.com/reference/android/view/View#findViewById(int)\">findViewById()</a>, and change nodes by calling methods like: <pre> button.setText(String) </pre> <pre>container.addView(View) </pre> <pre> img.setImageBitmap(Bitmap) </pre> These methods change the internal state of the widget. Not only can this be tedious, but updating views manually increases the likelihood of errors (e.g. forgetting to update a view).\n</p>\n<p>\nJetpack Compose is a fully declarative component-based approach, meaning you describe your UI as functions that transform data into a UI hierarchy. When the underlying data changes, the Compose framework automatically updates the UI hierarchy for you, making it simple to build UIs easily and quickly.\n</p>\n<p>\n\n<center> <iframe width=\"560\" height=\"315\" src=\"https://www.youtube.com/embed/SMOhl9RK0BA\" frameborder=\"0\" allow=\"accelerometer; autoplay; encrypted-media; gyroscope; picture-in-picture\" allowfullscreen></iframe></p> </center>\n\n\n<h2>Full interop with existing Android views</h2>\n\n\n<p>\nAdopting any new framework is a big change for existing projects and codebases, which is why we‚Äôve designed Compose to be as easy to adopt as Kotlin ‚Äî it is fully interoperable with existing Android code, from day one. \n</p>\n<p>\nMigrating to Compose depends on you and your team.
